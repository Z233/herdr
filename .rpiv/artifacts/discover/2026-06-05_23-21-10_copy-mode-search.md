# FRD: Copy mode search

## Summary
Add `/` (forward) and `?` (backward) search to herdr's copy mode, mirroring tmux behavior. While in copy mode, pressing `/` enters an incremental search sub-mode where the user types a query string; matches are found across the full scrollback history, the cursor jumps to the first match as the user types, all visible matches are highlighted on the pane, and `n`/`N` navigate between matches. Pressing Enter confirms the search (cursor stays at the match), while Esc cancels and restores the cursor to its pre-search position.

## Problem & Intent
I use herdr daily and need to search through pane scrollback/history quickly, like I do in tmux with `/`. Currently, copy mode only supports cursor movement, word/paragraph navigation, and easymotion-style two-character jumps — there is no way to find arbitrary text in the scrollback buffer. When I'm looking for a specific log line, error message, or command output that has scrolled off the visible viewport, I have to manually scroll through potentially thousands of lines. A search function would make this instant.

## Goals
- Enable searching for arbitrary text strings within the full scrollback history while in copy mode
- Match tmux's familiar `/` forward search and `?` backward search keybindings
- Provide immediate visual feedback: cursor jumps to first match as query is typed, visible matches are highlighted
- Support navigating between matches with `n` (next) and `N` (previous)
- Preserve the existing copy-mode workflow: Enter to confirm and stay at match, Esc to cancel and restore original cursor position

## Non-Goals
- Regex search (initial implementation uses plain string matching only)
- Configurable search keybinds (`/` and `?` are hard-coded vi-style, matching other copy-mode commands)
- Search across multiple panes simultaneously
- Persistent search history between copy mode sessions
- Case-sensitive search as the default (case-insensitive is the default; case toggle is out of scope)

## Functional Requirements
1. **FR-1 — Search entry**: Pressing `/` while in copy mode enters forward search sub-mode. Pressing `?` enters backward search sub-mode. The bottom status bar changes to show "SEARCH" mode label, the current query text, and hints (e.g., `n/N next  enter confirm  esc cancel`).
2. **FR-2 — Query accumulation**: While in search sub-mode, printable characters (alphanumeric, symbols, space) are appended to the search query. Backspace removes the last character. The query is displayed in the bottom bar.
3. **FR-3 — Incremental search**: After each character is added or removed, the system re-scans the scrollback history for matches. The first match (in search direction) becomes the active match; the cursor jumps to that match's row and column, and the viewport scrolls to bring it into view if necessary.
4. **FR-4 — Match highlighting**: All visible matches (matches on rows currently within the viewport) are highlighted with an accent-colored background or similar visual distinction. The active match (the one the cursor is on) uses a different highlight style (e.g., bold + accent) to distinguish it from other visible matches.
5. **FR-5 — Case-insensitive matching**: Search matching is case-insensitive by default. The comparison uses `to_lowercase()` or equivalent on both query and scrollback text.
6. **FR-6 — Full scrollback scan**: The search scans the entire scrollback buffer (all historical rows plus the current viewport), not just the currently visible rows. Rows are read using the existing `extract_selection` / `read_text_screen` path with screen-buffer coordinates.
7. **FR-7 — Wrap-around navigation**: Pressing `n` moves to the next match in the search direction. Pressing `N` moves to the previous match (opposite direction). If the end of the scrollback is reached, search wraps around to the beginning (for forward search) or vice versa (for backward search).
8. **FR-8 — Enter to confirm**: Pressing Enter while in search sub-mode exits search sub-mode but keeps the cursor at the current match position. The bottom bar returns to normal copy-mode hints. The user can then continue with copy-mode actions (yank, selection, etc.).
9. **FR-9 — Esc to cancel**: Pressing Esc while in search sub-mode exits search sub-mode and restores the cursor to its original row and column from before search entry. The viewport scrolls back to the pre-search position. The bottom bar returns to normal copy-mode hints.
10. **FR-10 — Empty query behavior**: If the query becomes empty (e.g., via backspace), no match is active and the cursor returns to its pre-search position. The viewport does not scroll.

## Non-Functional Requirements
- **Performance**: Search scan must complete within ~100ms for scrollback up to 10,000 lines. If scrollback is larger, scan incrementally or truncate to a reasonable maximum (e.g., last 50,000 lines).
- **Security**: No external input — search query is typed by the user in copy mode. No sanitization concerns.
- **UX / Accessibility**: Search overlay text must be readable against the panel background. Active match highlight must have sufficient contrast. The search mode label ("SEARCH") must be visually distinct from normal copy mode ("COPY").
- **Reliability**: If the pane's scrollback buffer is empty or no matches are found, the search sub-mode continues gracefully — the user can keep typing or press Esc to cancel. No panics or unrecoverable states.

## Constraints & Assumptions
- Copy mode uses viewport-relative `cursor_row`/`cursor_col` coordinates (`src/app/state.rs:928-936`). Search jumps must update these and call `sync_copy_mode_selection()` (`src/app/input/copy_mode.rs:612-660`).
- Search scans scrollback via the Ghostty terminal runtime's `extract_selection()` or `read_text_screen()` API. The scan iterates row-by-row using screen-buffer (absolute) coordinates.
- The `regex` crate is **not** in current dependencies; initial implementation uses standard string matching (`str::find()`, `str::contains()`).
- Search sub-mode follows the existing EasyMotion sub-mode pattern (`src/app/input/copy_mode.rs:177-243`): add `search: Option<CopyModeSearchState>` to `CopyModeState` (`state.rs:928`), intercept `/` and `?` in the command character dispatch (`copy_mode.rs:153`), route keys to a dedicated sub-handler when `search.is_some()`.
- Bottom bar overlay follows the existing `render_copy_mode_overlay()` pattern in `src/ui/menus.rs:63-120`.
- Search highlighting on the pane follows the existing `render_copy_mode_easymotion_labels()` / `render_copy_mode_cursor()` pattern in `src/ui/panes.rs:317-377`.

## Acceptance Criteria
- [ ] Entering copy mode and pressing `/` shows "SEARCH" in the bottom bar with an empty query and hints.
- [ ] Typing `error` in search mode causes the cursor to jump to the first occurrence of "error" in the scrollback (case-insensitive).
- [ ] All visible occurrences of the query are highlighted on the pane; the active match (where the cursor is) has a distinct highlight.
- [ ] Pressing `n` moves the cursor to the next match; pressing `N` moves to the previous match.
- [ ] Pressing Enter exits search mode and leaves the cursor at the current match; the bottom bar returns to "COPY" mode.
- [ ] Pressing Esc exits search mode and restores the cursor to its pre-search position; the viewport scrolls back to the pre-search offset.
- [ ] Pressing `?` instead of `/` searches backward (first match is the closest occurrence above the cursor).
- [ ] Search wraps around: pressing `n` at the last match jumps to the first match.
- [ ] Backspace removes the last query character and re-searches; empty query returns cursor to pre-search position.
- [ ] `cargo test` passes for new copy-mode search tests (entry, query accumulation, match jumping, cancel/restore, wrap-around).

## Recommended Approach
Follow the EasyMotion sub-mode pattern exactly: add `search: Option<CopyModeSearchState>` to `CopyModeState` in `src/app/state.rs` (alongside `easymotion`), intercept `/` and `?` in `handle_copy_mode_key()` at `src/app/input/copy_mode.rs:153`, accumulate query characters in a dedicated sub-handler, scan scrollback row-by-row using `extract_selection()` with screen coordinates, update `cursor_row`/`cursor_col` and call `sync_copy_mode_selection()` for jumps, and render search state in `render_copy_mode_overlay()` at `src/ui/menus.rs:63`.

**Rationale**: `evidence: src/ui/panes.rs:323-334` (easymotion label rendering), `src/ui/panes.rs:362-377` (cursor rendering) + confirmed

### Case sensitivity
**Question**: Should search be case-insensitive by default (like tmux), or case-sensitive?
**Recommended**: Case-insensitive by default
**Chosen**: Case-insensitive by default
**Rationale**: tmux default behavior; optimizes for quick finds when exact case is unknown

### Wrap-around
**Question**: Should search wrap around when reaching the end/beginning of scrollback?
**Recommended**: Wrap around (tmux default)
**Chosen**: Wrap around
**Rationale**: tmux default behavior; avoids manual direction switching for sparse matches

### Search UX behavior
**Question**: How should search behave as the user types?
**Recommended**: Incremental search with pane highlighting — cursor jumps to first match as you type, visible matches highlighted, Enter to confirm, Esc to cancel
**Chosen**: Incremental + highlight
**Rationale**: Matches tmux behavior; maximizes visual feedback and responsiveness
